# Contributing to Future Herd

Future Herd is in early stage development. No application code exists yet. At this point, the most valuable contributions are not code — they are the decisions and knowledge that will make the code worth writing.

If you have domain expertise in agricultural organization governance, participatory design, rural technology, civic tech, or related fields, your input now is worth more than pull requests later.

---

## What to Contribute Right Now

**Architecture decision records.** The significant technical and design decisions that will shape Future Herd have not all been made yet. ADRs document the options, the reasoning, and the choice. See the pending decisions listed in [docs/architecture/README.md](docs/architecture/README.md).

**Design critiques.** If you have experience with participatory governance platforms, opinion-mapping tools, or rural-facing technology, your critique of the design direction is valuable. Review the design documents in [docs/design/](docs/design/) as they emerge and open issues with your observations.

**Research references.** Relevant work on participatory governance, opinion-clustering algorithms, rural technology adoption, trust systems, and agricultural organization governance belongs in [docs/research/](docs/research/). A well-placed reference can save weeks of design work.

**Domain expertise.** If you work inside an agricultural organization and can speak to how decisions actually get made — how information flows, where participation breaks down, what leaders need — that context is foundational. Open an issue and share what you know.

---

## How to Propose an Architecture Decision

1. Open an issue using the **Architecture Decision** template.
2. Describe the decision to be made, the context, the options you see, your recommendation, and the reasoning.
3. Leave the issue open for discussion. Architecture decisions benefit from multiple perspectives before anything is locked in.
4. Once there is rough consensus, a pull request can be opened to add the ADR as a document in [docs/architecture/](docs/architecture/).

Do not open a pull request for an ADR before the issue discussion has happened.

---

## How to Contribute Design Work

Wireframes, user flow diagrams, and interaction designs go in [docs/design/](docs/design/). Open an issue first to describe the scope of what you want to contribute and align on format and approach before doing the work. Priority design areas are listed in [docs/design/README.md](docs/design/README.md).

---

## How to Contribute Research

Relevant research, case studies, and reference material go in [docs/research/](docs/research/). This includes existing work on:

- Participatory governance platforms and their adoption patterns
- Opinion-mapping algorithms (Polis and alternatives)
- Rural technology adoption
- Agricultural organization governance structures in Canada
- Trust and reputation systems (Stack Exchange, Wikipedia, etc.)
- Civic tech platforms and why they succeed or fail

If you have a reference that belongs here, open a PR with a brief summary and a link or citation. The bar for inclusion is relevance, not novelty.

---

## Code Contributions (Future)

Code contributions are not solicited yet. The architecture and data model need to be established first. When the project is ready for code, the tech stack, development environment, and coding conventions will be documented before contributions are opened up.

Watch the repository or check [docs/roadmap.md](docs/roadmap.md) for when this changes.

---

## Communication

Discussion happens in GitHub Issues and Pull Requests. If you are unsure where to start, open an issue and ask. There are no wrong questions at this stage.

As the community grows, additional communication channels may be established — including, eventually, using Future Herd itself once it is functional.

---

## Review Process

All contributions go through pull request review. At minimum, a PR needs one approving review before merge. Architecture decisions require discussion in an issue before a PR is opened. For research and design contributions, a brief issue describing the scope is encouraged but not required for smaller additions.

Reviewers will check that contributions are consistent with the project's design principles as described in [README.md](README.md).
